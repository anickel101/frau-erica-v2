# Frau Erica — API (api/)

AWS Lambda + API Gateway backend, deployed via AWS SAM. Serves two
things: read-only gated content (Persons/Families/search, behind
Cognito auth) and the account lifecycle (request access, admin
approval, login support). Public content (Documents/Galleries/Lexicon)
does **not** go through here — see the root `CLAUDE.md` and
`app/CLAUDE.md` for why (static export, no live backend needed).

Live stack: `frau-erica-api`, region `us-east-1`. Base URL is in
`template.yaml`'s `Outputs.ApiUrl` and hardcoded (non-secret) in
`app/src/config/cognito.ts`.

## Tech stack, and why

- **AWS SAM** (`template.yaml`) — CloudFormation, but with a much
  shorter syntax for the Lambda/API Gateway/IAM boilerplate this
  project is mostly made of. `sam build`/`sam deploy` are the whole
  deploy loop; `samconfig.toml` (committed — holds non-secret config
  only, see "Secrets" below) remembers the stack name/region/profile
  so `sam deploy` needs no flags after the first `--guided` run.
- **Lambda, Node.js 22, one function per route** — not one big
  function with internal routing. Matches this project's existing
  pattern (`GetPersonFunction`, `SearchFunction`, etc.) and keeps each
  function's IAM policy scoped to exactly what that one route needs
  (see "IAM philosophy" below).
- **esbuild** (via SAM's built-in `BuildMethod: esbuild`) — bundles
  each handler to a single JS file. `Format: cjs`, `Target: node22` on
  every function block.
- **API Gateway HTTP API**, not REST API — cheaper, simpler, sufficient
  for this project's needs. One Cognito JWT authorizer
  (`DefaultAuthorizer`) covers every route except `/request-access`
  (see "Auth" below).
- **sql.js** — the query layer reads a SQLite file via WASM, not
  `node:sqlite` (which `app/`'s local export script uses) or
  `better-sqlite3` (native bindings, a real cross-compile headache
  going Mac→Lambda-Linux). The compiled `.wasm` binary is embedded as a
  base64 string (`src/lib/sqlWasmBase64.ts`, generated by
  `scripts/generate-wasm-asset.ts`, regenerate after bumping the
  `sql.js` version) rather than shipped as a real file — SAM's esbuild
  build method has no clean way to ship a real binary alongside bundled
  JS; embedding sidesteps that entirely. See `src/lib/db.ts`.
- **Vitest** — same house style as `app/` (no semicolons, single
  quotes, trailing commas, see `.prettierrc.json`).

## How a request actually flows

1. API Gateway's Cognito JWT authorizer validates the token (issuer +
   audience) before the Lambda ever runs. This is a **401** concern —
   proves _a_ valid account is asking, nothing about _what_ they're
   allowed to see.
2. Inside the handler, `requireApprovedAccess()` or
   `requireAdminAccess()` (`src/lib/auth.ts`) checks the `cognito:groups`
   claim and short-circuits with a **403** if the caller isn't in the
   right group. This is page-type-level gating, not per-record — see
   `app/CLAUDE.md`.
3. Only then does the handler touch the database (`getDb()`,
   `src/lib/db.ts` — downloads the S3 snapshot on cold start, cached at
   module scope so warm invocations skip it) or call out to Cognito/SES/
   SSM.

`/request-access` is the one exception — it has no account to check yet
(`Auth: Authorizer: NONE` override in `template.yaml`), so it's a plain
`APIGatewayProxyEventV2`, not the JWT-authorizer variant every other
handler uses. It protects itself with reCAPTCHA v3 server-side
verification instead.

## Established conventions

- **Handler = thin, logic = in `lib/`.** Handlers (`src/handlers/*.ts`)
  parse the request, call the auth guard, delegate to `lib/queries/*.ts`
  or a `lib/*.ts` module, and format the response via
  `jsonResponse()` (`src/lib/response.ts`). Business logic lives in
  `lib/`, specifically so it's testable without an `event` object.
- **IAM philosophy: narrow, hand-written, no managed policies.** Every
  function's `Policies` block is a hand-written `Statement` scoped to
  exactly the actions and exactly the one resource ARN it needs — never
  a wildcard resource, never a broad managed policy like SAM's
  `S3ReadPolicy` (which grants whole-bucket access). The admin functions
  (`AdminApproveUserFunction` etc.) are the most sensitive grants in
  the stack — they can create real login credentials — and are scoped
  correspondingly tightly (only the specific `cognito-idp:Admin*`
  actions each one actually calls, on the one user pool ARN, nothing
  else).
- **Testing: mock AWS SDK clients as real `class` bodies, not
  `vi.fn()`.** The SDK constructs clients with `new`, so
  `vi.fn(() => ({...}))` fails with "is not a constructor" — mock as
  `class { send = sendMock }` instead. See any `*.test.ts` that mocks
  `@aws-sdk/client-*` for the exact pattern.
- **Authorization tests prove the short-circuit, not just the status
  code.** Every guarded handler's test mocks its AWS SDK client's `send`
  to _throw_ if called, then asserts a 403 with `send` never invoked —
  proving the guard runs before any side effect, not just that a 403
  eventually comes back. This is the single most important test in any
  new admin/gated handler.
- **Query-layer tests build a real SQLite database from the real
  schema** (`src/lib/testFixtures.ts`, using `schema/schema.sql`
  directly) rather than mocking query results — catches real SQL bugs,
  not just wiring bugs.
- **Secrets never go in `template.yaml`/`samconfig.toml`** (both are
  committed to git). Only non-secret config lives there (Cognito pool
  ID, API URLs, S3 bucket/key names). The one real secret this project
  has — the reCAPTCHA key — lives in SSM Parameter Store as a
  `SecureString`, fetched at Lambda runtime (`src/lib/secrets.ts`,
  cached at module scope like the DB snapshot). See "AWS services" below
  for exactly why.

## How to add a new route

1. Handler file in `src/handlers/`, following the existing pattern:
   parse input, call `requireApprovedAccess`/`requireAdminAccess` first
   if it's gated, delegate to `lib/`, return via `jsonResponse()`.
2. New `AWS::Serverless::Function` block in `template.yaml` — copy the
   shape of an existing one (`Metadata.BuildMethod: esbuild` block,
   `Handler`, `Policies` scoped to exactly what this route needs,
   `Events` with the `HttpApi` path/method). Only add `Auth: Authorizer: NONE`
   if the route genuinely has no account yet (like
   `/request-access`) — everything else inherits the pool-wide
   authorizer automatically.
3. Tests: a `lib/` unit test for any new pure logic, a handler test
   proving the auth guard short-circuits (if gated).
4. `npm run ci` (lint, typecheck, format, test) — then `sam build` to
   confirm the template/bundle are actually valid, not just that the
   TypeScript compiles.

## How to deploy

```
sam build && sam deploy
```

No flags needed after the first `--guided` run — `samconfig.toml`
remembers everything. Review the changeset SAM shows before confirming;
it should only ever show the resources you actually touched.

**Per this project's standing preference: AWS resource creation
(new Cognito setup, SES identity verification, IAM users, etc.) is done
by the user directly, guided step-by-step — not executed by Claude.**
`sam deploy` itself is an exception the user has run directly multiple
times once the template is ready; the guided/user-run rule mainly
applies to console-driven, one-off AWS setup (verifying an SES identity,
registering reCAPTCHA, creating an IAM user), not routine code deploys
once established.

## How to test locally

- `npm run ci` — lint, typecheck, format check, and the full Vitest
  suite. No AWS credentials needed; every AWS SDK call is mocked.
- `sam build` then `sam local start-api` — needs Docker running (Lambda
  runtime emulation) and real-ish AWS credentials if a handler actually
  needs to reach a real AWS service (the Cognito authorizer itself is
  _not_ evaluated locally by `sam local` — routes just run unauthenticated).
- For anything that needs to be proven against the real, deployed stack
  (auth actually enforcing, an email actually arriving, a Cognito
  account actually landing in the right group) — there's no substitute
  for hitting the real API with `curl` and a real Cognito token. This
  project's own history has repeatedly caught real bugs this way that a
  green `sam build` alone would have missed (an `esbuild external`
  packaging assumption that would have shipped a broken Lambda; a
  `DesiredDeliveryMediums` default that would have silently sent
  invitations nowhere). Trust a live check over an assumption.

## AWS services used, and why

| Service                      | Used for                                                                                                                                                                                                                                                                                                | Why this one                                                                                                                                                                                                                                    |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Lambda**                   | All compute — the 8 functions in `template.yaml`                                                                                                                                                                                                                                                        | Pay-per-request, near-zero cost for a low-traffic family site; matches the project's original plan over an always-on server                                                                                                                     |
| **API Gateway (HTTP API)**   | Routing, the Cognito JWT authorizer, CORS                                                                                                                                                                                                                                                               | Cheaper and simpler than REST API; this project has no need for REST API's extra features                                                                                                                                                       |
| **Cognito**                  | User accounts, `pending`/`approved`/`admin` groups, the `custom:person_id` link to the family database, SRP login, password reset, and the account-creation invitation email                                                                                                                            | Fully managed auth with groups and custom attributes built in — avoids hand-rolling password storage/reset/session handling                                                                                                                     |
| **S3**                       | `frau-erica-db-backups` (nightly SQLite snapshots + a `current/` pointer the Lambda reads); `frau-erica-images-*` (public photos, fronted by CloudFront, not touched by `api/` directly)                                                                                                                | Durable, cheap object storage; the natural place to stage a periodically-refreshed read-only DB snapshot for Lambda to pull                                                                                                                     |
| **CloudFront**               | CDN in front of the image S3 bucket (OAC-locked — CloudFront is the only path in, direct S3 access 403s)                                                                                                                                                                                                | Real photos need real caching/edge delivery; also lets the bucket stay fully private                                                                                                                                                            |
| **SES**                      | The Request Access admin-notification email (`src/lib/ses.ts`)                                                                                                                                                                                                                                          | Already the natural AWS choice for sending mail from a Lambda; the single verified identity (`FrauErica.archivist@gmail.com`, both sender and recipient) stays comfortably inside the sandbox tier's limits — no need for SES production access |
| **SSM Parameter Store**      | The one real secret in this project — the reCAPTCHA secret key, as a `SecureString`                                                                                                                                                                                                                     | CloudFormation can't create `SecureString` parameters directly, and a template-level secret reference would leak the plaintext into the Lambda's environment variables anyway; runtime fetch + cache keeps it out of both git and the console   |
| **KMS**                      | Implicitly, to decrypt that `SecureString` (`alias/aws/ssm`, the default key)                                                                                                                                                                                                                           | Comes bundled with SSM `SecureString` — `kms:Decrypt` is a separate, easy-to-forget IAM grant alongside `ssm:GetParameter`                                                                                                                      |
| **IAM**                      | A role per Lambda function (auto-created by `sam deploy` from each function's `Policies` block); a deploy user (`frau-erica-v2-deploy`, broad access, used only from the developer's machine); a separate backup user (`frau-erica-backup-user`, S3-write-only, used only by the nightly backup script) | Least-privilege per function; deliberately never reusing the backup user's narrow credentials for deploy work or vice versa                                                                                                                     |
| **CloudFormation** (via SAM) | The entire `api/` stack as infrastructure-as-code                                                                                                                                                                                                                                                       | Reproducible, reviewable infra changes instead of manual console clicks; `sam` is just a thinner syntax over this                                                                                                                               |

**Not used**: Route53 or ACM (no custom domain/certificate yet — the
API is used at its default `*.execute-api.amazonaws.com` URL; revisit
once real hosting/domain work happens), SES production access (sandbox
tier is sufficient at this project's real email volume), any
Enterprise/paid tier of anything.

## Known limitations / open items

- No production frontend origin exists yet — `template.yaml`'s
  `FrontendOrigin` parameter (used for CORS and the Request Access
  email's deep link) defaults to `http://localhost:5173`. Revisit once
  hosting is actually planned (the project is deliberately staying
  local-only for now — see the root-level project memory, not this
  file, for that decision).
